ember: a cpu-first llm inference engine in rust
voidwest
may 2025
abstract
we present ember, a lightweight inference engine for gpt-2 class transformer language models implemented in approximately 2,200 lines of rust. ember runs quantized models on cpu without external framework dependencies, loading models in the gguf v3 binary format. the system achieves 12 tok/s decode throughput on a 124m-parameter gpt-2 model via a pre-allocated kv cache and allocator-free hot path. the entire transformer architecture is generic over a Backend trait, enabling future gpu acceleration without modification to model logic. we describe the architecture, key design decisions, encountered failure modes, and quantitative performance characteristics.

1. introduction

deploying transformer-based language models on consumer hardware typically requires execution environments spanning hundreds of thousands of lines: pytorch, onnx runtime, or the llama.cpp ecosystem. these systems provide generality at the cost of complexity, making them difficult to audit, modify, or port to resource-constrained targets.

ember takes the opposite approach. it is a single-purpose inference engine: gpt-2, gguf format, cpu-first, and under 2,200 lines of rust. every tensor allocation is explicit. the inference hot path performs zero heap allocations. the transformer is generic over a Backend trait, decoupling model code from execution hardware.

the primary contribution is not performance — scalar matmul cannot compete with simd-accelerated frameworks on throughput. rather, it is a demonstration that a complete transformer inference pipeline, from binary model parsing through token generation, can be expressed in a compact, auditable codebase without sacrificing correctness or extensibility.

2. system architecture

ember is organized into eight modules, each responsible for a single stage of the inference pipeline: 

main.rs            entry point, cli args, generation loop
├─ loader.rs       gguf v3 parser, tensor loading + dequant
├─ model.rs        gpt-2 transformer blocks
│  ├─ backend.rs   backend trait + cpu backend impl
│  ├─ tensor.rs    row-major f32 tensor with basic ops
│  └─ kv_cache.rs  flat key/value cache for incremental decode
├─ sampler.rs      temperature, top-k, top-p sampling
├─ tokenizer.rs    huggingface tokenizer wrapper
└─ quant.rs        q8_0 block dequantization

2.1 the backend trait

the central abstraction is the Backend trait. every transformer component — attention, mlp, layer norm — is generic over B: Backend. there is one concrete implementation: CpuBackend, a zero-sized struct that delegates to CpuTensor methods running in scalar cpu loops. the trait abstracts linear algebra operations (matmul, add_broadcast), element-wise functions (gelu, softmax), normalization (layer_norm), and tensor lifecycle (zeroes, load_from_cpu, data, shape).

limitation. attention forward methods call data() to extract raw f32 slices and run attention math in scalar cpu loops, bypassing the backend. a gpu backend would still execute attention on the cpu through this path. adding fn attention(...) to the trait is planned future work; the trait is documented as honest about what it currently covers.

2.2 tensor primitives

CpuTensor is row-major, backed by a flat Vec<f32>. every operation allocates a new tensor. no views, no mutation in place. the cost is additional allocations; the benefit is that every operation is trivially correct and the compiler can reason about ownership. this proved critical during debugging (section 4).

2.3 gguf format handling

gguf v3 is a self-describing binary format comprising a 4-byte magic number (0x46554747), version field, metadata key-value pairs, and a tensor info table specifying name, dimensions, data type, and byte offset for each weight tensor. ember supports f32 and q8_0 quantized tensors. q8_0 uses 8-bit block quantization with an fp16 scale per block of 32 values, achieving approximately 4× compression.

a critical implementation detail concerns storage order: gguf stores q8_0 and f16 tensors in column-major order so the innermost dimension aligns with the block size (32). the tensor info header reports the logical shape in row-major convention, but the raw bytes are laid out column-major. failing to account for this transposition scrambles every weight matrix (see section 4.1).

3. inference pipeline

3.1 softmax

softmax is implemented in two passes. the first pass computes the maximum value to subtract for numerical stability: exp(x_i - max) / sum(exp(x - max)), which prevents overflow at exp(88). an edge case occurs during early decode when causal masking sets every element in a row to -inf. standard softmax then produces NaN ((-inf - -inf).exp() evaluates to NaN per ieee 754). our implementation detects this case and returns a uniform distribution of 1/n per position.

3.2 activation and normalization

gelu (gaussian error linear unit) is used as the activation function in the mlp block: 0.5 * x * (1 + erf(x / √2)). the implementation uses libm::erff for portability. layer normalization is applied per row as (x - mean) / √(var + ε) with learned scale and bias, with ε = 10⁻⁵. forgetting to assign the normalized result back to the activation variable produces silently degraded output — no panic, no NaN, just progressively worse text quality.

3.3 causal attention

attention is computed as standard scaled dot-product attention with a causal mask. query, key, and value projections are extracted from a single linear layer c_attn: ℝ⁷⁶⁸ → ℝ²³⁰⁴, packed as contiguous slices. the scaling factor 1/√d_head is precomputed once per forward pass. the causal mask sets future positions to -inf, which becomes 0.0 after softmax exponentiation.

3.4 kv cache

without caching, generating N tokens after a prompt of length P requires N+P full attention passes, each O(L²) where L is the current sequence length. the kv cache eliminates this redundancy by storing key and value projections for every token at every layer.

during prefill, the full prompt is forwarded through all 12 layers and every position's k and v are stored. during decode, only the single new token's k and v are computed and appended; attention reads the full history from cache. the cache is a flat [layer][head][pos][head_dim] array of ~72 mb (12 layers × 12 heads × 2048 positions × 64 dimensions × 4 bytes × 2 for k and v), pre-allocated once at initialization. a scratch buffer for attention scores is likewise pre-allocated and reused, achieving zero heap allocations in the inference hot path.

3.5 sampling

logit sampling supports temperature scaling, top-k truncation, and nucleus (top-p) sampling. categorical sampling uses inverse cdf over the filtered distribution. when temperature is zero, deterministic argmax is used, producing identical output on every run — critical for testing correctness against a reference implementation.

4. failure modes and debugging

4.1 column-major layout (bug #4)

symptom. deterministic greedy sampling (temperature 0) produced different output on every run. the same gguf file loaded in llama.cpp produced coherent text.

root cause. gguf stores q8_0 and f16 tensors in column-major order. for a weight matrix shaped [768, 50257], data on disk is laid out column-by-column: all 768 rows of column 0, then all 768 rows of column 1. the loader dequantized the flat buffer and called reshape(&[768, 50257]), which assumes row-major. every weight matrix in the model was transposed.

fix. reverse the dimension vector in the loader before reshaping for column-major tensor types. correlation with a pytorch reference forward pass improved from 0.004 to 0.999975.

4.2 kv cache prefill overwrite (bug #5)

symptom. prompt "1, 2, 3, 4, 5" produced "/5/5/5/5/5/5/5/". the model attended to at most one prompt token — the last one.

root cause. KVCache::append wrote every entry at self.cursor, which is deliberately not advanced until all 12 layers complete. during prefill, the attention loop called append for each prompt token, all landing at cursor = 0. only the final token's k/v survived.

fix. append now takes an explicit pos: usize parameter. the attention forward pass snapshots the cursor and passes cursor + pos for each token.

5. design decisions

three architectural choices proved immediately correct:

  1. transposed embeddings on load. gguf stores embeddings as [vocab, embed]. the loader transposes them so index_select performs a contiguous memcpy rather than a strided gather, at the cost of one transpose at startup.
  2. uniform softmax for all-masked rows. the all--inf branch costs one comparison per row and prevents NaN from propagating through 12 layers into the output logits.
  3. #[must_use] on tensor ops. since all tensor operations allocate new storage, dropping the return value silently discards the computation. #[must_use] caught a missing assignment in the layer norm block loop at compile time, preventing a silently-degraded model.

6. limitations and future work

  • scalar matmul. the current implementation delegates to matrixmultiply::sgemm, a pure-rust scalar implementation. simd vectorization (avx2 on x86, neon on arm) is projected to yield 4–8× throughput improvement, implemented via a SimdBackend that satisfies the existing Backend trait.
  • gpt-2 only. the gguf loader matches tensor names against hardcoded gpt-2 identifiers. llama-family support requires four additional primitives: rmsnorm, silu activation, rotary position embeddings (rope), and grouped-query attention (gqa). each is specified with starter code and tests.
  • attention on cpu. the Backend trait does not currently abstract attention, so even a gpu backend would execute attention in scalar cpu loops. adding fn attention(...) to the trait is planned.
  • not no_std. file i/o and mmap require the standard library. an embedded target (arm cortex-m) would need a no_std variant with pre-loaded weights.
